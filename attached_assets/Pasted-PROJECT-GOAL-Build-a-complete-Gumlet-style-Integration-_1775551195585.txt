PROJECT GOAL
Build a complete “Gumlet-style Integration API Module” inside my existing Video CMS so that any LMS can connect to the CMS using a simple API + SDK integration and play secure videos without tightly coupling LMS and CMS pages.

This must be a production-ready architecture, cleanly implemented, database-backed, and fully integrated into my CURRENT CMS codebase.

IMPORTANT CONTEXT
This is my current Video CMS architecture. You must implement this feature INSIDE the existing codebase, not as a disconnected demo.

Current stack and architecture:
- Frontend: React 18 + TypeScript + Vite
- Backend: Express 5 + TypeScript
- ORM: Drizzle ORM
- Database: Postgres / Supabase
- Video delivery: HLS playback via hls.js
- Existing secure flow already exists with:
  - /api/player/:publicId/mint
  - /api/player/:publicId/manifest
  - /hls
  - /seg
  - /key
  - playback sessions
  - embed tokens
  - session rotation
  - analytics
  - global/per-video security
  - watermark/banner/security settings
- CDN / secure path layer uses Cloudflare Worker + SIGNING_SECRET + ORIGIN_BASE
- Existing architecture already has tables and logic for:
  - videos
  - video_player_settings
  - video_banners
  - video_watermark_settings
  - video_security_settings
  - embed_tokens
  - playback_sessions
  - audit_logs
  - system_settings
  - video_client_security
  - storage_connections
  - media_assets

TARGET PRODUCT
Turn my Video CMS into a reusable “Video Delivery Platform” for LMS, like Gumlet:
- CMS is the secure video backend
- LMS is the external client
- LMS should only need:
  1. an integration client key
  2. a backend-signed launch token
  3. a small JS embed script or React wrapper
- Then LMS can render and control player behavior easily
- CMS handles secure playback, session minting, tracking, playback policy, and secure manifest/segment/key delivery

MAIN DELIVERABLES
Implement all of the following completely:

1. DATABASE LAYER
Create all required database tables and relations using Drizzle ORM migrations and schema updates.

2. BACKEND API LAYER
Create a new Integration API module with secure external endpoints for LMS clients.

3. SERVER-SIDE SECURITY MODEL
Implement full trust chain:
- LMS backend signs launch token
- CMS validates integration client + signature
- CMS mints internal playback/embed token
- Only CMS-issued token can access manifest/segments/keys
- Support origin restrictions, expiry, user binding, and session logging

4. ADMIN PANEL

5. SDK / EMBED LAYER

6. LMS INTEGRATION CONTRACT

7. ANALYTICS + SESSION TRACKING

8. CLEAN ARCHITECTURE
No hacks, no duplicated insecure logic, no breaking existing CMS player functionality.

ABSOLUTE RULES

PHASE 3 — LAUNCH TOKEN MODEL
==================================================

Implement a proper LMS → CMS launch token model.

SUPPORTED MODE 1: HMAC SIGNED PAYLOAD
The LMS backend signs a payload using the client secret.

Expected launch token payload:
- iss: integration client key or slug
- aud: "cms-player"
- sub: LMS user id
- publicId: video public id
- courseId: optional
- lessonId: optional
- sessionId: optional
- name: optional
- email: optional
- permissions: optional object
- startAt: optional
- origin: optional LMS origin
- exp: unix timestamp required
- iat: unix timestamp required
- jti: unique token id required

Implement validation rules:
- must not be expired
- publicId required
- sub required
- jti required
- exp required
- iat required
- token issuer must map to active integration client
- optional origin must match allowed_origins if strict check enabled
- video must be allowed for that client if selected-mode enabled

IMPORTANT:
This launch token is ONLY for minting.
It must NOT be accepted at manifest/hls/seg/key endpoints.

1) POST /api/integrations/player/:publicId/mint
Purpose:
Main LMS entry point. Accept launch token, validate it, create launch log, create integration playback session, mint CMS embed token, return manifest URL + config.

Request body:
{
  "launchToken": "....",
  "context": {
    "courseId": "...",
    "lessonId": "...",
    "sessionId": "...",
    "origin": "https://lms.example.com"
  }
}

Behavior:
- verify :publicId matches token publicId
- validate client
- validate origin if required
- validate video exists and is published/playable
- resolve permissions and player config
- call or reuse current internal mint/session creation logic cleanly
- create integration_launch_logs entry
- create integration_playback_sessions entry
- mint short-lived CMS embed token
- return response

Response shape:
{
  "ok": true,
  "integrationSessionId": "...",
  "embedToken": "...",
  "expiresIn": 300,
  "manifestUrl": "https://.../api/player/:publicId/manifest?token=...",
  "refreshUrl": "/api/integrations/player/:publicId/refresh",
  "pingUrl": "/api/integrations/player/:publicId/ping",
  "eventUrl": "/api/integrations/player/:publicId/events",
  "metadata": {
    "title": "...",
    "durationSeconds": 1234,
    "posterUrl": "...",
    "publicId": "..."
  },
  "playerConfig": {
    ...
  }
}

2) POST /api/integrations/player/:publicId/refresh
Purpose:
Refresh short-lived embed/playback token for active integration session.

Request body:
{
  "integrationSessionId": "...",
  "embedToken": "..."
}

Behavior:
- verify active session
- verify session still valid
- verify publicId matches
- optionally map to internal playback session refresh logic
- return new embed token and manifest URL
