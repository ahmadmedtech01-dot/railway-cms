Replit Agent Prompt — Apply Same Advanced Security to LMS Playback Flow

Problem:
Advanced video security is working on public/shareable links, but when the same video plays inside my LMS iframe via postMessage LMS_LAUNCH_TOKEN, the same security layers are not applied.

Current LMS flow:
- Admin Embed & Share tab gives iframe code without token in URL
- LMS generates/fetches HMAC launch token server-side
- LMS sends token to iframe using postMessage type: LMS_LAUNCH_TOKEN
- CMS creates playback session
- Video plays inside LMS

But security settings like stealth protected mode, MediaSource/appendBuffer guard, server-gated playlist, short TTLs, download velocity scoring, strong heartbeat, key binding, watermark anti-tamper, and session revocation must apply here too.

Important:
Do NOT break:
- shareable links
- /v/:publicId
- /watch/:shareCode
- /s/:shareCode
- admin preview
- LMS postMessage launch flow
- existing embed tokens
- existing LMS integration clients
- upload/transcode/storage
- B2/R2/S3 hiding
- /hls /seg /key or stream/chunk/secret pipeline

Goal:
Create ONE unified playback bootstrap/session pipeline used by all entry points:
1. share link
2. public /v link
3. LMS iframe postMessage
4. admin preview

All flows must receive and enforce the same security settings.

Required changes:

1. Unify playback bootstrap logic

Find all places that create playback sessions/tokens:
- share link bootstrap
- /v/:publicId bootstrap
- /watch/:shareCode bootstrap
- LMS postMessage token validation
- admin preview token bootstrap

createSecurePlaybackSession({

  userId,
  lmsClientId,
  lmsUserId,

  referer,

  userAgent,
  adminPreview,
  shareCode
})

This function must always load:
- global security settings
- per-video security override
- watermark settings
- player settings
- advanced hardening settings

And return the same secure playback config to frontend:
- playbackSessionId
- short-lived player token
- stream mode
- stealthProtectedPlaybackMode
- mediaSourceGuardEnabled
- appendBufferGuardEnabled
- serverGatedPlaylistWindowEnabled
- shortTokenTtlsEnabled
- strongHeartbeatV2Enabled
- downloadVelocityScoringEnabled
- strongerKeyBindingEnabled

- security event endpoint

- violation thresholds

- validate token signature
- validate nonce + exp replay protection
- validate allowed LMS origin/client
- validate video access
- then call the same createSecurePlaybackSession function used by share links

The LMS flow must NOT use a legacy/light session path.

The LMS session must enforce:
- stealth protected playback mode
- opaque chunk routes if enabled
- MediaSource/appendBuffer guard
- strong heartbeat v2
- server-gated playlist window
- download velocity scoring
- stronger key binding
- short token TTLs
- watermark anti-tamper
- session revocation

3. Ensure frontend applies config from LMS bootstrap

In embed-player.tsx or LMS player init logic:
- after receiving valid LMS_LAUNCH_TOKEN, call the backend bootstrap/launch endpoint
- use returned security config to initialize player
- do not fallback to insecure/default HLS mode if advanced settings are enabled
- start MediaSource guard before hls.js initializes
- start strong heartbeat v2 if enabled
- use opaque stream/chunk/secret routes if stealth mode is enabled
- apply watermark anti-tamper
- send security events to returned endpoint

Important:
If security config is missing, fail closed for protected videos. Do not silently fallback to insecure mode.
