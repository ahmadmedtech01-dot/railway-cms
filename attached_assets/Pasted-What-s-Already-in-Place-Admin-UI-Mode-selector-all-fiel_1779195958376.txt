The Actual Gap (2 small changes)
Gap 1 — Backend endpoint always puts token in URL (line 600):

embedUrl: `${cmsBase}/embed/${vid.publicVideoId}?embedToken=${encodeURIComponent(embedToken)}`

This works for query_param mode but defeats postMessage mode (token leaks into page source). It also doesn't return the tokenMode value so the player can't branch.

Gap 2 — Student player blindly uses that URL (line 334):

setEmbedUrl(data.embedUrl);  // always has token in it

It never checks tokenMode, never sends LMS_LAUNCH_TOKEN via postMessage.

Proposed Minimal Change
Backend (server/routes/video-public.ts, ~10 lines):

Add tokenMode to response (from vid.tokenMode, defaulting to 'postMessage')
Add bareEmbedUrl: ${cmsBase}/embed/${vid.publicVideoId} (no token)
Keep embedUrl field as-is (token in URL) for backward compat with query_param
Student player (StudentTopicVideoPlayer.tsx, ~30 lines):

After fetching token, branch on tokenMode:
query_param → use data.embedUrl (current behavior, no change)
postMessage → use data.bareEmbedUrl as iframe src, store embedToken in a ref, post {type:'LMS_LAUNCH_TOKEN', token} to iframe on PLAYER_READY (plus a retry loop, mirroring the admin preview pattern at VideoTab.tsx:129-168)
No schema changes. No new endpoints. No admin UI changes. Query Param mode continues working identically.
