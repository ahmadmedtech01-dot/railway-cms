========================================================
PROBLEM:

- Your detection logic counts normal HLS behavior (player fetching playlist + segments + key) as “too many requests”

- Keys endpoint is counted too aggressively
- Rate limits are global (shared across users) rather than per user/IP/session/video

This prevents preview from being blocked.

B) Fix false positives for real users:
- Apply detection per (ip + videoId + sessionId) instead of global.
- Ignore normal HLS pattern:
  - Allow sequential segment fetches.
  - Allow 1–2 parallel segment requests (players often prefetch).
- Do not count playlist refreshes too harshly (m3u8 can be polled).
- Key endpoint should allow reasonable retries; key spamming threshold must be higher.

C) Smarter thresholds:
Implement a sliding window with separate buckets:
- playlistRequests: allow e.g. 30/min
- segmentRequests: allow e.g. 240/min (4/sec average) per user+video
- keyRequests: allow e.g. 20/min (keys sometimes re-requested)
Block only if:
- segment parallelism > N (e.g. >6 concurrent) OR
- segmentRequests > huge threshold (e.g. >600/min) OR
- keyRequests > threshold AND segmentRequests are also high (strong signal)
Add a “cooldown” shorter (e.g. 10–20 sec) for mild limits; keep 60 sec only for extreme.

D) Do not punish normal rebuffering / seeking:
- Seeking causes burst of segment requests—whitelist short bursts:
  allow a burst buffer: +50 segments within 10 seconds, once every 60 seconds.

E) Add logs + admin visibility:
- Log why blocked: which bucket exceeded, counts, ip, videoId
- In admin preview, if something is blocked, show message “Security blocked: reason…” but it should not happen after bypass.
